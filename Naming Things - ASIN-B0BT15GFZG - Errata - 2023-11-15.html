The use of i and j for loop indices, p and q for pointers, and s and t for strings is so frequent that there is little profit and perhaps some loss in longer names”. [4]
Note - 6. Understandability > Page 29
The single-letter names for pointers and strings maybe a relic of the past. Right now compilers have excellent diagnostics that can protect against handling types incorrectly. So nowadays it probably makes much more sense to use descriptive names for pointers and strings. Except in some special cases like when they're only used locally and only in a handful of lines. In this case trying to give them names may force other important variables (e.g., parameters) to have more complex names.

Highlight (yellow) - 7. Conciseness > Page 41
In many cases, the metadata about an identifier (e.g., its type, its intended scope) can be understood without reading its name, so including information about the type within the name is redundant and adds unnecessary length and noise. Identifier metadata should not be included in names.
Note - 7. Conciseness > Page 42
The metadata in the name also has the negative effect of duplication which can cause other issues.
Highlight (yellow) - 7. Conciseness > Page 45
Names of booleans are sometimes prefixed with “not” (e.g., not_prepared) or a negative prefix (e.g., invalid), which has two undesirable effects: it 1) increases the length of the name without adding any meaning and 2) creates the opportunity for confusing double negatives within conditionals. Boolean names should always be stated in the positive to avoid these issues. # Bad if !user.is_invalid(): user.save # Good if user.is_valid(): user.save
Note - 7. Conciseness > Page 46
A case could be made that one factor is how the logic is built. For example, if there's always a guard for the invalid case then it'd be better to use the negative form and avoid having to negate the positive form everywhere.
