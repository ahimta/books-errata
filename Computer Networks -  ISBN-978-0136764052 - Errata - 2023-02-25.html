Highlight (yellow) - 8.13 Social Issues > Page 846
These new remailers, often called cypherpunk remailers, work as follows. The user produces an email message, complete with RFC 822 headers (except Fr om:, of course), encrypts it with the remailer’s public key, and sends it to the remailer.
Note - 8.13 Social Issues > Page 846
Should mention that single re-mailers still have some serious issues, which are quite intuitive. For example, a compromized re-mailer that receives from the original sender directly can still save logs. This can happen either intentionally by the server owner or by a cracker who would make the server send it the logs. So both trust in the server's owner and the absence of crackers are something to be considered. For a chain of re-mailers, the same applies but more parties have to be compromized. One caveat is that only compromizing the first and last maybe sufficient. Another point, that maybe reasonable to not mention here, is that chain re-mailers can be less secure in the field since their operators may percieve the overall system to be more secure or assume incorrectly that operators of other servers' security can make up for their mistakes (diffusion of responsibility).
